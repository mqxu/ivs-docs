需求规格说明书

1. 引言

1.1 编写目的

本需求规格说明书的编写旨在明确“智能车载系统”软件开发的功能性和非功能性需求，为系统设计、开发、测试、验收和后期运维提供标准依据。本文档面向项目管理者、开发人员、测试团队、用户代表及相关方，确保各方对需求理解一致，为后续项目归档与评优提供权威资料。

1.2 项目背景与目标

随着智能网联汽车和新一代移动互联网技术的融合发展，车载系统日益向智能化、集成化和数据化演进。行业急需能够支持多角色、多端访问、智能分析和高安全性的车载管理平台。本项目以“智能车载系统”为目标，开发集车辆管理、能耗分析、远程充电、可视化大屏于一体的移动应用平台，支持车主、管理员、运营方等不同用户需求，提升行业智慧运营能力，实现行业可持续、高质量发展。

1.3 定义

  • 智能车载系统：指基于移动终端与云端平台，实现车辆远程管理、数据分析、充电调度等功能的综合性软件系统。
  • 用户：包括车主（个人/企业）、管理员、运营方三类用户。
  • 充电桩平台：第三方能源服务系统，可为本系统提供充电点数据与充电控制接口。
  • 其他专业术语参见《GB/T 5271 信息技术词汇》和项目词汇表。

1.4 参考资料

  • GB/T 9385-2008 软件需求规格说明书编写指南
  • 相关行业标准：GB/T 25000.51（软件工程 产品质量要求）、GB/T 8567-2020（计算机软件文档编制规范）
  • 智能网联汽车行业应用案例、主流移动应用开发技术白皮书等。

2. 总体描述

2.1 产品概述

本系统为智能车载行业开发，基于 UniApp/Vue3 多端开发框架，SpringBoot 后端微服务，MySQL 数据库和 ECharts 可视化。系统采用前后端分离架构，支持车主 App、小程序、管理后台和大屏端，提供车辆状态、远程充电、能耗大屏、消息推送、数据统计、权限管理等综合功能。

2.2 用户角色与主要业务场景

  • 车主：实时查询车辆状态、远程控制、查找充电桩、扫码充电、历史记录分析。
  • 管理员：车辆/用户管理、数据统计、运营监控、权限维护、批量导入导出。
  • 运营方：能耗分析、故障监控、数据大屏、系统集成与行业创新。

2.3 运行环境

  • 前端：安卓/iOS 设备、Web、主流小程序平台；
  • 后端：Linux/Windows 服务器，Java 17+，SpringBoot，MySQL 8+；
  • 可视化：大屏显示器、支持 ECharts；
  • 对接服务：高德/腾讯地图 API、主流充电桩平台 API。

2.4 设计/实现约束

  • 符合国家与行业信息安全及数据合规要求；
  • 接口须遵循 RESTful 标准，支持自动化文档和 Mock 调试；
  • 前后端分离、松耦合、可扩展、支持云部署与本地部署。

2.5 假定与依赖

  • 地图与充电桩第三方平台稳定可用；
  • 终端设备具备正常网络与定位能力；
  • 相关法规、行业政策无重大调整。

3. 功能需求

3.1 功能模块总览

功能编号功能名称适用角色说明
F1车辆状态管理车主、管理员实时状态、告警、维护
F2用户绑定与管理车主、管理员注册、实名认证、解绑
F3充电服务车主、管理员地图找桩、扫码充电
F4历史数据与统计全部角色行驶/充电/故障历史
F5可视化大屏管理员、运营方多维数据展示
F6消息推送与告警车主、管理员实时推送与自定义告警
F7后台管理与权限管理员、运营方多角色管理、数据归档
F8系统对接与扩展管理员、运营方API 对接、第三方平台

3.2 各功能描述

F1 车辆状态管理

  • 描述：系统通过硬件或接口实时采集车辆数据（如位置、电量、故障码），前端以图表/地图/文本等方式呈现，支持车辆状态监控与异常报警。
  • 输入：车载终端数据、用户操作请求
  • 输出：实时状态数据、状态变更推送
  • 处理逻辑：定时拉取与 WebSocket 实时推送相结合，异常则触发告警和数据记录。

F2 用户绑定与管理

  • 支持手机号/微信注册、车辆绑定与解绑，实名认证流程，管理员可批量审核、维护账号。
  • 权限控制、数据一致性校验，异常操作日志留痕。

F3 充电服务

  • 用户通过地图检索充电桩，扫码启动充电，实时获取充电状态和记录；后端需与第三方平台对接，安全处理充电流程。

F4 历史数据与统计

  • 多维度查询行驶、充电、故障历史，按时间、车辆等条件筛选，支持图表导出与报表归档。

F5 可视化大屏

  • 运营与管理大屏展示车辆分布、能耗排行、运营趋势，支持数据钻取和多维分析。

F6 消息推送与告警

  • 充电完成、异常故障等消息自动推送，支持用户自定义订阅与静默设置。

F7 后台管理与权限

  • 支持多级角色分配、车辆和用户批量管理、操作日志、材料归档导出。

F8 系统对接与扩展

  • 预留标准 API，支持地图、充电桩、能耗分析等第三方平台接入，接口遵循 RESTful，支持鉴权和回调。

4. 性能需求

  • 支持不少于 1000 台车辆同时在线，车辆数据刷新周期 ≤3 秒；
  • 关键接口响应时间 ≤2 秒，系统核心模块高可用，99.99%在线；
  • 前端页面在主流设备首屏加载 ≤2 秒，数据同步/推送延迟 ≤5 秒。

5. 数据需求

  • 车辆、用户、充电等核心数据建模，关系型结构（E-R 图/数据字典归档）；
  • 所有数据支持按用户/时间/业务类型多条件筛选、聚合与导出；
  • 关键数据字段加密存储，支持异地备份。

6. 接口需求

6.1 API 要求

  • 所有 API 采用 RESTful 标准，支持 GET/POST/PUT/DELETE；
  • 返回 JSON 格式，接口文档自动生成（Swagger/OpenAPI），配 Mock 数据集成；
  • 关键接口（如充电操作、绑定解绑）强制鉴权，采用 OAuth2/JWT 安全机制。

6.2 第三方集成

  • 集成高德/腾讯地图 API（定位、POI 检索、路径导航）；
  • 集成主流充电桩运营平台 API，包含充电启动、状态回调、历史记录、数据同步；
  • 可扩展接入能耗分析或行业创新服务。

7. 安全性要求

  • 用户注册/登录/操作全链路加密传输，存储敏感数据需加密；
  • 多角色访问控制，操作日志全程留痕、可追溯；
  • 防范常见安全威胁（SQL 注入、XSS、CSRF 等）。

8. 兼容性与可移植性

  • 支持主流安卓/iOS 设备、微信/支付宝等小程序平台、主流 Web 浏览器；
  • 后端服务可云部署或本地部署，支持 Docker/K8s 一键部署；
  • 数据迁移与备份机制完善，支持行业升级与扩展。

9. 可靠性与可维护性

  • 系统模块化设计，支持分模块热更新与运维；
  • 异常自诊断与自动报警，日志自动归档、支持远程维护；
  • 接口与业务逻辑注释齐全，便于二次开发与材料评优。

10. 易用性

  • UI/UX 设计简洁清晰，操作流程可视化引导；
  • 地图、充电等高频功能一键直达，支持智能推荐和新手指引；
  • 提供帮助中心、FAQ、在线反馈通道。

11. 需求变更与追踪机制

  • 设立需求变更管理流程：需求提出 → 评审 → 归档 → 开发 → 测试 → 验收 → 复盘；
  • 每条需求分配唯一编号，全部变更、状态、归档于需求管理平台（支持 Excel/Notion/JIRA 等追踪）；
  • 所有历史变更可溯源对比，确保开发与验收全流程有据可查。

12. 其他要求

  • 所有需求规格、流程、接口、材料须支持项目归档、赛项申报与企业复审；
  • 关键流程、结构、接口配 Mermaid/PlantUML 等标准流程图或结构图归档，便于后续查阅和材料归档。
