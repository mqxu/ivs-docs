UML 系统建模

一、用例图（核心业务流程建模）

1. 主要用例覆盖

  • 车辆管理
  • 用户注册/绑定
  • 充电服务
  • 历史数据分析
  • 消息通知与告警
  • 后台管理与权限控制

2. 用例图（PlantUML 示例）

@startuml
actor 车主
actor 管理员
actor 运营方

usecase "车辆信息管理" as UC1
usecase "用户绑定/解绑" as UC2
usecase "扫码充电" as UC3
usecase "历史数据分析" as UC4
usecase "消息通知" as UC5
usecase "后台管理" as UC6

车主 --> UC1
车主 --> UC2
车主 --> UC3
车主 --> UC4
车主 --> UC5

管理员 --> UC1
管理员 --> UC6
管理员 --> UC4
管理员 --> UC5

运营方 --> UC6
运营方 --> UC4
@enduml

图形说明与归档

  • 用例图应随业务调整动态维护，归档于“系统需求文档”相关章节。
  • 建议用 PlantUML/StarUML 生成图片，配用例简述表（功能-角色-描述）。

二、类图（系统核心实体关系建模）

1. 核心实体

  • 车辆（Vehicle）
  • 用户（User）
  • 充电桩（ChargingPile）
  • 充电记录（ChargeRecord）
  • 消息通知（Notification）
  • 管理员（Admin）
  • 运营方（Operator）

2. 类图（PlantUML 示例）

@startuml
class Vehicle {
  +vehicleId: String
  +plateNumber: String
  +status: String
  +ownerId: String
}

class User {
  +userId: String
  +name: String
  +phone: String
  +role: String
}

class ChargingPile {
  +pileId: String
  +location: String
  +status: String
}

class ChargeRecord {
  +recordId: String
  +vehicleId: String
  +pileId: String
  +startTime: Date
  +endTime: Date
  +energy: Double
}

class Notification {
  +notificationId: String
  +userId: String
  +content: String
  +sendTime: Date
}

User "1" -- "0..*" Vehicle
User "1" -- "0..*" ChargeRecord
Vehicle "1" -- "0..*" ChargeRecord
ChargingPile "1" -- "0..*" ChargeRecord
User "1" -- "0..*" Notification

@enduml

归档与维护建议

  • 类图应纳入“系统架构设计文档”与“数据库 ER 图”配套维护。
  • 每次数据库、对象模型调整后需同步更新类图，标明版本号。

三、时序图（关键交互场景）

1. 典型交互（如扫码充电）

@startuml
actor 用户
participant "移动端" as Frontend
participant "后端" as Backend
participant "充电桩平台" as PilePlatform
participant "数据库" as DB

用户 -> Frontend: 打开App扫码
Frontend -> Backend: 提交充电请求
Backend -> PilePlatform: 下发充电指令
PilePlatform -> Backend: 返回充电状态
Backend -> DB: 记录充电过程
Backend -> Frontend: 状态反馈与进度
Frontend -> 用户: 显示充电信息
@enduml

维护建议

  • 常见交互流程（如用户绑定、车辆上报、异常告警等）均需建模，版本迭代时及时补充归档。

四、活动图/流程图（复杂业务流程）

1. 预约充电活动图（Mermaid 示例）

flowchart TD
    A[用户发起预约充电] --> B{检查车辆/用户状态}
    B -- 合格 --> C[显示可用充电桩]
    C --> D[用户选择桩并预约]
    D --> E[后端校验并锁定桩位]
    E --> F[预约成功，消息推送]
    B -- 不合格 --> Z[流程中止，提示修正]

2. 地图查找流程图

flowchart LR
    A[进入找桩页面] --> B[地图加载当前位置]
    B --> C[检索附近充电桩]
    C --> D[筛选/排序]
    D --> E[点击桩点查看详情]
    E --> F[发起导航/充电]

维护建议

  • 流程发生重大变更时及时补图、归档，建议统一编号、配流程说明文本。

五、建模工具推荐

  • PlantUML（开源、可集成至 VSCode/IDEA、易自动化脚本生成）
  • Mermaid（适合文档、Markdown 与网页快速嵌入）
  • StarUML（专业 UML 设计工具，支持复杂工程与团队协作）
  • 其他：Visual Paradigm、Enterprise Architect（如需企业级管理）。

六、图形与模型描述及维护建议

  • 所有 UML 图形须统一归档至设计/需求文档专章，按业务/功能/时间顺序编号与标注版本。
  • 图形需配简要功能说明与变更记录，便于评优、查阅和材料迭代。
  • 推荐开发/设计团队约定建模模板和注释规范，定期校验图文一致性。
  • 重要场景可配“用例-类-时序-流程”全链路建模，助力归档、评审与 AIGC 自动文档生成。
