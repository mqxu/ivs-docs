一、典型业务场景端到端流程

1. 用户扫码充电全流程

场景描述

车主通过 App 地图页选中充电桩，扫码完成车辆与桩的匹配，后端验证后启动充电，过程全程状态反馈，充电结束自动生成账单并推送通知。

步骤详解

  1. 地图页面展示充电桩分布，车主选择目标桩点
  2. 点击“扫码充电”按钮，打开摄像头扫描二维码
  3. 前端发起 /api/charge/start 请求，带上车辆 ID、桩 ID、token
  4. 后端校验车辆、桩状态及权限，写入充电记录，推送充电开始状态
  5. 前端订阅 WebSocket 充电状态流，实时展示剩余时间/功率
  6. 充电完成后，后端生成账单，推送充电完成和费用通知
  7. 用户可在充电历史页/消息中心查看本次充电详情

2. 轨迹回放数据流转

场景描述

车主选择指定车辆及日期，系统查询历史轨迹点和能耗，地图可视化展现行驶轨迹与能耗变化。

步骤详解

  1. 用户在历史数据页选择车辆与时间区间
  2. 前端请求 /api/track/history 获取轨迹点数据
  3. 后端从 vehicle_status、energy_data 表聚合对应数据
  4. 前端利用 ECharts/地图组件动态绘制轨迹及能耗趋势
  5. 用户可导出 PDF/Excel 轨迹报告

二、前端页面与后端服务联调方案

  • 接口一致性：严格遵循 API 文档，参数、数据结构保持同步
  • 本地 Mock：前端可用 Mock.js/Apifox 模拟后端接口，独立调试页面交互
  • 联调环境：配置开发/测试环境 API 网关，前端自动切换真实接口
  • WebSocket 联调：前端与后端约定主题/消息格式，便于充电等实时推送测试

三、各环节技术点与实现提示

轨迹回放ECharts、地图组件前端路径渲染、能耗叠加、交互体验优化

五、联调测试脚本与自动化验证建议

  • 接口自动化：用 Postman/Apifox 编写端到端接口集合，覆盖扫码、启动、进度、结束、历史数据等全链路流程，断言关键字段和返回码。
  • WebSocket 测试：编写前端/Node 小脚本模拟消息订阅与推送，自动验证实时推送有效性。
  • Mock/真实数据切换：配置 Mock/真实环境一键切换，确保所有环节可独立/集成测试。
  • 异常/边界测试：覆盖桩点离线、车辆未绑定、充电中断、网络异常等负面场景。

六、Demo 演示与归档方式

  • 演示材料：录制端到端流程视频（屏录+旁白）、配 PPT 说明流程节点和技术细节。
  • 代码归档：Demo 源码单独归档，含 README 和部署启动说明，便于评审和查阅。
  • 测试报告：联调、自动化测试结果、Bug 记录和优化建议归档，便于复盘。
  • 复现环境：建议提供本地一键部署脚本（如 docker-compose、mock server），方便外部评委/老师快速演示。
